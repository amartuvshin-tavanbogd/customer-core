Өмнө нь: банк зөвхөн банкны дотоод гүйлгээ харж байсан — Дулмаагийн гүйлгээ тийм ч олон биш тул татгалздаг байсан.

Customer Core-тэй болсноор:

Finance систем Дулмаагийн нэгдсэн профайлд хандаж дараах зүйлийг харна:
  • Нура: 3 жил, 24 удаа шинжилгээ — бүгдийг хугацаанд төлсөн
  • Finance хадгаламж: 1.2 сая₮ тогтмол хадгаламж — хоцроолт тэг

Эдгээр нь Дулмаа найдвартай, тогтмол төлбөр хийдэг хүн гэдгийг нотолно.

0 секунд — дата алдагдалгүй

Point-in-time Recovery

≤ 4 цаг — сэргэлтийн хугацаа

05 — ТОГТВОРТОЙ БАЙДЛЫН ПАТЕРНҮҮД
Нэг систем тасрахад нөгөө нь ажиллаж байх механизм
Finance болон Нура-гийн интеграцид хэрэглэгдэх 4 архитектурын паттерн.
🔌
Circuit Breaker
Нура систем тасрахад Finance Customer Core-д хандаж чадахгүй болно. Circuit Breaker нь энэ тохиолдолд fallback горимд шилжинэ — Finance дотоодын скор дангаараа ашиглана. Нура сэргэмэгц автоматаар дахин холбогдоно.
Горимууд: CLOSED (хэвийн) → OPEN (тасарсан, fallback) → HALF-OPEN (сэргэж байна)
📬
Outbox Pattern
Finance зээл олгоод Customer Core-д мэдэгдэх ёстой. Хэрэв мэдэгдэл явуулж чадахгүй бол яах вэ? Outbox Pattern: эхлээд локал дата санд бичиж, дараа нь Kafka-д дамжуулна. Алдаа гарсан ч дата алдагдахгүй.
Баталгаа: Exactly-once delivery. Database transaction + Kafka publish атомик.
🔄
Saga Pattern
Зээл олгох нь Finance + Нура 2 систем нэгэн зэрэг шинэчлэгдэх ёстой distributed transaction. Нэг нь амжилтгүй бол compensating transaction — буцаан хийгдэх ажиллагаа автоматаар эхлэнэ.
Зээл олгох Saga: Reserve credit → Notify Nura → Confirm → [ Fail? → Compensate ]
📭
Dead Letter Queue (DLQ)
Нура-гийн Kafka message 3 удаа retry хийгээд амжилтгүй бол DLQ-д орно. Инженер мэдэгдэл авч, шалтгааныг шинжлэж гараар засна. Жишээ нь: мессежийн format өөрчлөгдсөн.
Дараалал: Kafka topic → Consumer retry x3 → DLQ → Alert → Manual fix → Replay
